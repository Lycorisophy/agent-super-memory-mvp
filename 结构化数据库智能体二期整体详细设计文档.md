## 1. 概述

### 1.1 背景与一期衔接

一期已实现：表结构文档（Markdown / Excel）与自然语言（LLM）入库、`schema_docs` 向量检索、Neo4j `Table` / `Column` / `HAS_COLUMN`、可选 **`RELATES_TO`**（由解析器从「关联」列或 LLM JSON 写入）、`/schema/query` 单步生成只读 SQL 及 `dry_run` / `auto_execute`，以及基于 Ollama 的 **ReAct 多步查询**（`agent_mode=react`）。

典型一期局限：复杂问题依赖「一次看清所有候选表 + 一条 SQL」；多跳推理、执行失败后自动修正、按需逐步探查表结构等场景覆盖不足。

**二期定位**：在一期基础设施之上，以「可解释的多步推理 + 可演进的知识与工具链」为主线，提升复杂查询成功率、可观测性与企业落地能力；与一期 API **向后兼容**。

### 1.2 二期设计目标（总览）

| 维度 | 目标说明 |
| :--- | :--- |
| **查询编排** | 默认保留单步模式；增强 ReAct 的提示词、工具契约、步数与 SQL 次数配额、失败重试与 trace 结构化输出。 |
| **知识获取** | 统一表信息提取管线（规则 + LLM）：Markdown / Excel / 自然语言多源、字段键与关联列、入库幂等与 Neo4j 图一致。 |
| **图与检索** | 完善 `RELATES_TO` 语义与 `get_foreign_keys` 消费场景；可选图遍历补全 JOIN 候选。 |
| **安全与审计** | 维持只读白名单；增加 ReAct 每步审计日志、可选请求级 trace 落盘与脱敏策略。 |
| **数据源扩展** | MySQL **库表内省**（高置信度 schema）、可选 Java / ORM 元数据接入（规划）。 |
| **体验** | 多轮对话上下文、会话级 schema 子集缓存、错误信息对模型友好封装。 |

### 1.3 技术选型（在一期基础上的增量）

| 组件 | 一期选型 | 二期增量 / 可选 |
| :--- | :--- | :--- |
| 编排 | 自研 ReAct 循环（`schema_react_agent`） | 可选引入 **LangGraph** 做显式状态机与检查点；或保持自研以降低依赖。 |
| LLM | Ollama 单轮 / 多轮 JSON 协议 | 可选 **结构化输出**约束（JSON Schema）、小模型专用于 tool-router。 |
| Neo4j | 社区版 + `Column.column_key` UNIQUE | 文档与实现统一；企业版可选复合约束策略另述。 |
| 观测 | 日志 | **OpenTelemetry** 或文件 trace 插件（规划）。 |

## 2. 系统架构（二期视角）

```
┌─────────────────────────────────────────────────────────────────┐
│                         用户 / 调用方                            │
└────────────────────────────┬────────────────────────────────────┘
                             │ HTTP
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                      FastAPI 服务层                              │
│  POST /schema/ingest   POST /schema/query                       │
│       │                        │                                 │
│       ▼                        ├─ agent_mode=single_pass ──▶ 单步 query_schema
│  SchemaManager                 │                                 │
│  （解析、Milvus、Neo4j）        └─ agent_mode=react ────────▶ ReAct 循环        │
└────────────────────────────┬────────────────────────────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        ▼                    ▼                    ▼
   Milvus                Neo4j                 MySQL
 schema_docs      Table/Column/          只读执行
                  RELATES_TO
        │                    │                    │
        └────────────────────┴────────────────────┘
                             │
                             ▼
                        Ollama（嵌入 + 对话）
```

### 2.1 与一期文档的差异说明（实现已对齐部分）

- **Neo4j 约束**：社区版不支持 `(Column.table_name, name)` 的 **NODE KEY**；实现采用 **`Column.column_key` + UNIQUE**，设计文档表述以仓库实现为准。
- **`/schema/query` 请求体**：`mode` 仅表示 `dry_run` / `auto_execute`；**`agent_mode`** 区分 `single_pass` / `react`；**`max_steps`** 控制 ReAct 最大步数。
- **入库**：支持 **仅自然语言**（`user_text` 足够长且无 `file`）由 LLM 产出 JSON 表数组后入库；文件路径仍支持 MD / XLSX 及表头中英列名、关联列解析。

## 4. ReAct 查询子系统（二期核心）

### 4.1 行为摘要（与实现对齐）

- **入口**：`agent_mode=react` 时由 `schema_react_agent.run_schema_react` 驱动。
- **工具**：`search_tables`、`describe_table`、`execute_sql`、`get_foreign_keys`、`final_answer`。
- **约束**：`execute_sql` 单次请求内调用次数上限（如 8）；全路径复用 `is_safe_sql`；`mode=dry_run` 时 `execute_sql` 仅校验不连库执行。
- **输出**：`final_answer`、`trace`（每步 thought / tool / arguments / observation）、`sql_tool_calls`、`steps_used`。

### 4.2 二期增强方向（设计项）

1. **提示词与 JSON 协议**：减少模型输出非 JSON；失败时自动压缩上下文或切换「仅 JSON」短提示。
2. **工具扩展**：`sample_rows`（受表白名单与行数限制）、`explain_sql`（MySQL EXPLAIN，仍受安全策略约束）。
3. **反思策略**：对同一 `execute_sql` 连续失败定义退避（改写次数上限、降级为只描述不执行）。
4. **可观测性**：结构化日志字段：`request_id`、`step`、`tool`、`latency_ms`、错误摘要（不含完整 SQL 中的敏感字面量策略可配置）。

## 5. 表信息提取子系统（二期与实现对齐）

### 5.1 输入源

| 类型 | 说明 |
| :--- | :--- |
| Markdown | `##` 分块；表头支持中英列名；可选「键」「关联」列；`目标表.目标列` 生成 `RELATES_TO` 候选。 |
| Excel | Sheet 名即表名；必须列与可选列的中英映射；「关联关系」列同 Markdown 规则。 |
| 自然语言 | LLM 输出严格 JSON 数组；解析失败重试一次；默认置信度建议 0.5、`source_type=nl_llm`。 |

### 5.2 统一结构化模型

与《表信息提取详细设计方案》一致：`name`、`comment`、`columns[]`、`relations[]`（`type`、`target_table`、`source_cols`、`target_cols`）；表名与字段名 **snake_case 规范化**。

### 5.3 入库与向量文本

- Milvus 文档正文包含 **「关联关系」** 段落，便于语义检索 JOIN 意图。
- Neo4j：`MERGE (a:Table)-[:RELATES_TO {rel_key}]->(b:Table)`，属性含 `type`、`source_cols`、`target_cols`、`left_col` / `right_col`（首列快捷字段，便于旧查询兼容）。

## 6. 安全设计（延续 + 强化）

- **SQL**：仅只读前缀 + 禁词扫描 + 禁止多语句；ReAct 内每次 `execute_sql` 均校验。
- **账号**：生产环境强制只读 MySQL 账号；禁止在文档中提交生产密码。
- **配额**：`max_steps`、`execute_sql` 次数上限、结果行数上限（已实现部分）可配置化到 `config` / 环境变量。

---

## 7. MySQL 库表内省（二期规划）

- 从 `INFORMATION_SCHEMA` 读取表、列、主键、外键（若存在）。
- 映射到与文档解析相同的 JSON 结构，置信度默认 **0.8**，`source_type=db_introspect`。
- 与人工文档冲突时的合并策略：**版本时间戳优先**或 **人工覆盖开关**（待产品确认）。

## 8. 多轮对话与上下文（二期规划）

- **会话对象**：`session_id` + 服务端存储「最近 N 步 trace 摘要 + 用户已确认的表集合」。
