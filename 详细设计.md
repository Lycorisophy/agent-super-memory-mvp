## 1. 概述

### 1.1 背景与目标

传统大模型记忆方案通常将事件、事实、知识分开存储，导致字段冗余、格式割裂、人类难以直接阅读数据库内容。本系统采用 **Milvus（向量库）+ Neo4j（图数据库）** 混合架构，并设计了**统一存储格式**，使事件、事实、知识共享同一套简洁的结构化中文模板。

设计目标：
- 统一事件、事实、知识的存储格式，减少冗余字段。
- 保留 Neo4j 图数据库的关系推理能力。
- 保留 Milvus 向量库的语义检索能力。
- 提供对大模型友好的 Function Calling 工具接口。
- 数据库内容自解释，降低调试和维护成本。
- 支持对时态和置信度的可选结构化过滤。

### 1.2 技术选型

| 组件 | 选型 | 作用 |
| :--- | :--- | :--- |
| 向量数据库 | Milvus 2.x | 存储语义向量，支持相似度检索 |
| 图数据库 | Neo4j 5.x | 存储记忆节点及关系，支持图游走 |
| 嵌入模型 | Ollama（默认 `qwen3-embedding:8b`，4096维） | 文本向量化 |
| 唯一ID生成 | 雪花算法 | 全局唯一、趋势递增 |
| 开发语言 | Python 3.10+ | 服务实现 |
| Web框架 | FastAPI | HTTP API 服务 |

### 1.4 v4/v5 与旧版不兼容说明

- **Milvus**：集合字段为 `memory_id`、`user_id`、`text`、`embedding`、`timestamp`、`memory_type`（默认集合名 `memory_vectors`）。旧版 `memory_chunks`（含 `chunk_id`/`ref_type`/`ref_id` 等）不可混用。
- **Neo4j**：使用 `:Memory` 节点与 `[:HAS_MEMORY]` 关系；旧版 `:Event`/`:Fact`/`:Knowledge` 相关约束与数据不再由本代码写入。
- **嵌入模型**：默认使用 `qwen3-embedding:8b`，向量维度 **4096**。若改用其他模型（如 `nomic-embed-text` 768维），须同步修改 `OLLAMA_EMBED_MODEL` 与 `VECTOR_DIM`，并重建 Milvus 集合。


## 2. 统一记忆格式定义

### 2.1 模板结构

所有记忆条目（事件、事实、知识）均存储为以下固定格式的中文文本：

```
【类型】{事件|事实|知识}
【时间】{YYYY-MM-DD HH:MM}
【地点】{地点描述，无则留空}
【主体】{主要关联人，无则留空}
【内容】{核心描述}
【来源】{原始对话片段或推断依据，可选}
```

**字段说明**：

| 字段 | 含义 | 填写说明 |
| :--- | :--- | :--- |
| `【类型】` | 记忆类别 | 必须为 `事件`、`事实` 或 `知识` |
| `【时间】` | 发生时间或记录时间 | 格式 `YYYY-MM-DD HH:MM`，若无法确定则使用当前时间 |
| `【地点】` | 事件发生地点或事实/知识的关联地点 | 无则留空 |
| `【主体】` | 执行动作的人或事实/知识的关联人 | 无则留空 |
| `【内容】` | 记忆的核心描述 | 事件：动作 + 结果；事实：键 = 值；知识：标题 + 要点 |
| `【来源】` | 原始依据 | 尽量保留用户原话或推理依据，便于溯源 |

### 2.2 示例

**事件示例**：
```
【类型】事件
【时间】2026-04-21 14:30
【地点】沃尔玛超市
【主体】我
【内容】与李丽一起购买排骨，买到了新鲜的排骨
【来源】昨天和李丽去超市买了排骨
```

**事实示例**：
```
【类型】事实
【时间】2026-04-21 14:35
【地点】
【主体】我
【内容】配偶姓名 = 李丽
【来源】用户说“我老婆叫李丽”
```

**知识示例**：
```
【类型】知识
【时间】2026-04-21 14:40
【地点】
【主体】
【内容】煮饺子防破技巧：水里加盐
【来源】用户分享的烹饪经验
```

### 2.3 格式优势

| 方面 | 说明 |
| :--- | :--- |
| **存储紧凑** | 每条记忆仅一个文本字段，无需多个属性列。 |
| **人类可读** | 打开数据库即可理解内容，便于调试和审计。 |
| **LLM友好** | 召回后直接作为上下文，无需二次解析。 |
| **扩展容易** | 新增记忆类型只需使用新的 `【类型】` 值，无需修改 Schema。 |


## 3. 数据库详细设计

### 3.1 Neo4j 图数据库

#### 节点设计

| 标签 | 属性名 | 中文说明 | 类型 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| `:User` | `user_id` | 用户唯一标识 | 字符串 | 必填 |
| `:Memory` | `memory_id` | 记忆唯一标识 | 字符串 | 雪花ID，必填 |
| | `content` | 统一格式的完整文本 | 字符串 | 见第2节模板，必填 |
| | `timestamp` | Unix时间戳（秒） | 整数 | 必填 |
| | `memory_type` | 记忆类型 | 字符串 | `event`、`fact`、`knowledge` |
| | `fact_key` | 事实键名 | 字符串 | 仅事实类型填充，可选 |
| | `fact_value` | 事实键值 | 字符串 | 仅事实类型填充，可选 |
| | `tense` | 时态（v5新增） | 字符串 | `past` / `present` / `future`，可选 |
| | `confidence` | 置信度（v5新增） | 字符串 | `real` / `imagined` / `planned`，可选 |

#### 关系设计

| 关系名 | 中文含义 | 方向 | 说明 |
| :--- | :--- | :--- | :--- |
| `[:HAS_MEMORY]` | 拥有记忆 | 用户 → 记忆 | 每个记忆必属于一个用户 |
| `[:NEXT]` | 顺序发生 | 记忆 → 记忆 | 前一事件紧接着后一事件 |
| `[:CAUSED]` | 导致 | 记忆 → 记忆 | 原因事件导致结果事件 |
| `[:SUB_EVENT_OF]` | 子事件属于 | 记忆 → 记忆 | 子事件是父事件的一部分 |
| `[:RELATED]` | 相关 | 记忆 → 记忆 | 通用关联关系 |
| `[:OVERRIDES]` | 覆盖 | 记忆 → 记忆 | 新事实覆盖旧事实（仅用于事实类型） |

**关系创建约束**：
- `NEXT`、`CAUSED`、`SUB_EVENT_OF` 通常仅用于 `memory_type='event'` 的节点之间。
- `OVERRIDES` 仅用于 `memory_type='fact'` 的节点之间，由系统自动维护。

#### 约束

```cypher
CREATE CONSTRAINT user_id_unique IF NOT EXISTS FOR (u:User) REQUIRE u.user_id IS UNIQUE;
CREATE CONSTRAINT memory_id_unique IF NOT EXISTS FOR (m:Memory) REQUIRE m.memory_id IS UNIQUE;
```

### 3.2 Milvus 向量库

**集合名称**：`memory_vectors`

| 字段名 | 数据类型 | 中文说明 |
| :--- | :--- | :--- |
| `memory_id` | 字符串(64) | 对应 Neo4j 中 Memory 节点的唯一标识 |
| `user_id` | 字符串(64) | 用户唯一标识 |
| `text` | 字符串(4096) | 用于向量化的文本（直接取 `content` 字段） |
| `embedding` | 浮点向量(4096) | 文本的语义向量（维度取决于嵌入模型） |
| `timestamp` | 64位整数 | Unix时间戳（秒） |
| `memory_type` | 字符串(32) | 记忆类型：`event`、`fact`、`knowledge` |

**索引配置**：
- 向量索引：`HNSW`，度量类型 `COSINE`，参数 `M=16, efConstruction=200`。
- 标量索引：对 `user_id`、`memory_type` 建立索引以加速过滤查询。

**注意**：v5 新增的 `tense`、`confidence` 字段**不存入 Milvus**，仅用于 Neo4j 侧的结构化过滤。原因：时态与置信度属于元数据属性，对语义相似度贡献极小；统一文本已自然包含这些信息，向量检索本身已能覆盖大部分需求；保持 Milvus Schema 精简可减少索引开销。


## 4. 雪花标识生成器

### 4.1 设计说明

采用雪花算法变体生成 64 位长整型唯一标识：
- **41 位时间戳**（毫秒级，起始时间 2024-01-01）
- **10 位机器标识**（5 位数据中心ID + 5 位工作节点ID）
- **12 位序列号**（单毫秒内递增）

特点：全局唯一、趋势递增、高性能本地生成。

### 4.2 Python 实现

```python
import time
import threading

class SnowflakeIDGenerator:
    def __init__(self, datacenter_id: int = 1, worker_id: int = 1):
        self.datacenter_id = datacenter_id & 0x1F
        self.worker_id = worker_id & 0x1F
        self.sequence = 0
        self.last_timestamp = -1
        self.lock = threading.Lock()
        self.twepoch = 1704067200000  # 2024-01-01 00:00:00

    def _current_millis(self) -> int:
        return int(time.time() * 1000)

    def _wait_next_millis(self, last_timestamp: int) -> int:
        timestamp = self._current_millis()
        while timestamp <= last_timestamp:
            timestamp = self._current_millis()
        return timestamp

    def next_id(self) -> int:
        with self.lock:
            timestamp = self._current_millis()
            if timestamp < self.last_timestamp:
                raise RuntimeError("时钟回拨异常")
            if timestamp == self.last_timestamp:
                self.sequence = (self.sequence + 1) & 0xFFF
                if self.sequence == 0:
                    timestamp = self._wait_next_millis(self.last_timestamp)
            else:
                self.sequence = 0
            self.last_timestamp = timestamp
            return ((timestamp - self.twepoch) << 22) | \
                   (self.datacenter_id << 17) | \
                   (self.worker_id << 12) | \
                   self.sequence

    def next_str(self) -> str:
        return str(self.next_id())
```

**标识前缀约定**：
- Neo4j 记忆节点标识：`mem_{雪花ID}`
- Milvus 向量条目标识：直接使用 `mem_{雪花ID}` 作为 `memory_id`。

## 7. 核心实现要点

### 7.1 存储流程

1. **接收参数**：`memories` 数组，每个元素包含 `type`、`time`、`location`、`subject`、`content`、`source`，以及可选的 `tense`、`confidence`。
2. **生成模板文本**：按模板拼接为 `content` 字符串。
3. **生成唯一标识**：为每条记忆生成 `mem_{雪花ID}`。
4. **写入 Neo4j**：
   - 创建 `Memory` 节点，属性 `memory_id`、`content`、`timestamp`、`memory_type`，事实类型额外填充 `fact_key`、`fact_value`。
   - 将 `tense`、`confidence`（若提供且合法）写入节点。
   - 与用户节点建立 `HAS_MEMORY` 关系。
5. **处理事实覆盖**：若为事实类型，查找同 `fact_key` 且值不同的旧节点，自动创建 `(新)-[:OVERRIDES]->(旧)` 关系。
6. **写入 Milvus**：将 `content` 向量化后插入 `memory_vectors` 集合。
7. **处理关系**：根据 `relations` 数组中的临时ID映射，在对应 `Memory` 节点间创建关系。

### 7.2 查询流程

1. **向量检索**：对 `query_text` 向量化，在 Milvus 中检索候选 `memory_id`，可按 `memory_type` 和时间范围过滤。若传入了 `tense`/`confidence`，候选数自动放大以补偿后续过滤损失。
2. **Neo4j 过滤与回填**：
   - 根据候选 `memory_id` 列表，在 Neo4j 中按 `tense`、`confidence`（若提供）二次过滤。
   - 对于事实类型，优先返回未被 `OVERRIDES` 覆盖的最新节点。
3. **图扩展（可选）**：沿 `CAUSED` 关系双向游走，构建因果链摘要。
4. **返回结果**：将 `content` 文本列表及因果链返回给大模型。

### 7.3 混合检索职责划分

| 步骤 | 负责组件 | 输入 | 输出 | 时态/置信度参与方式 |
| :--- | :--- | :--- | :--- | :--- |
| 语义粗筛 | Milvus | `query_text` | 候选 `memory_id` 列表 | 不参与 |
| 精确过滤 | Neo4j | `memory_id` + `tense`/`confidence` | 最终结果 | 精确匹配 |

### 7.4 为什么 `tense` 和 `confidence` 不存入 Milvus

- **语义贡献低**：时态与置信度对向量相似度影响极小（如“梦见中彩票”与“中了彩票”向量距离极近），无法靠向量精确区分。
- **减少索引开销**：保持 Milvus Schema 仅有 6 个核心字段，降低内存和磁盘占用。
- **职责分离**：Milvus 管“像不像”，Neo4j 管“是不是”，各司其职。
- **统一文本已隐含**：`content` 模板中的自然语言描述（如“梦见”“计划”）已携带时态与置信度信息，向量检索本身可覆盖大部分需求。

## 9. 部署与运行

```bash
pip install -r requirements.txt
```

### 9.2 启动基础服务

```bash
# Milvus
docker run -d --name milvus-standalone -p 19530:19530 milvusdb/milvus:latest

ollama pull qwen3.5:35b-a3b-q8_0
```

## 10. 附录

### 10.1 v5.0 相对 v4.0 增量

| 方面 | v4.0 | v5.0 |
| :--- | :--- | :--- |
| Neo4j `Memory` 节点属性 | 4个基础属性 | 新增 `tense`、`confidence` 可选属性 |
| `store_memory` 参数 | 无时态/置信度 | 每项可带 `tense`、`confidence` |
| `query_memory` 参数 | 无过滤 | 可选 `tense`、`confidence` 过滤 |
| 返回结果 | 无时态/置信度字段 | 节点有值时附带 `tense`/`confidence` |

### 10.2 项目结构

| 文件 | 作用 |
| :--- | :--- |
| `main.py` | FastAPI 入口 |
| `config.py` | 配置管理 |
| `memory_service.py` | Milvus + Neo4j + 嵌入核心逻辑 |
| `memory_agent.py` | Ollama 编排 + 系统提示词 |
| `tools_spec.py` | 工具 JSON Schema |
| `test_main.http` | HTTP 测试用例集 |

### 10.3 扩展建议

- **事件自动摘要压缩**：当连续小事件过多时，可触发大模型生成粗粒度父事件，建立 `SUB_EVENT_OF` 关系。
- **知识缓存**：对高频知识查询结果进行 Redis 缓存。
- **实体类型细化**：在 `【内容】` 中通过命名实体识别增强实体关联。
